# Taint Engine (Step 4) Implementation Plan

**Shared conventions:** [Shared Contracts and Naming Conventions](../reference/2026-06-27-shared-contracts-and-conventions.md)
**Logging:** [Logging Conventions](../reference/2026-06-27-logging.md)
**Error handling:** [Error Handling Conventions](../reference/2026-06-27-error-handling.md)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build `apps/taint` — a standalone, independently runnable, rule-driven taint/dataflow analyzer that consumes `ast-index/`, `facts.json`, and one external YAML rule file, then writes shared `FindingsDoc` JSON.

**Architecture:** `apps/taint` exposes analyzer-owned YAML rule models in `com.oversecured.sast.taint.model`, a library API (`TaintAnalyzer`) for orchestrator/in-process use, and a picocli `taint` CLI. The engine loads rules, canonicalizes source/sink/sanitizer signatures through `SignatureParser`, walks `AstIndex` method bodies, performs flow-sensitive intra-procedural propagation with simple CFG nodes, adds bounded method summaries, models Intent-extra ICC by key, applies sanitizer and reachability filters, and emits analyzer-agnostic `FindingsDoc`/`Finding`. Scope stays challenge-realistic: no full IFDS/IDE, no precise alias/points-to analysis, no reflection, no native, no implicit-flow tracking.

**Tech Stack:** Java 17; Gradle Groovy DSL; JUnit 5 + AssertJ; Jackson databind + YAML; picocli; JavaParser AST APIs through `com.oversecured.sast.parser.AstIndex`; shared `common` contracts (`ManifestFacts`, `SignatureParser`, `MethodSignature`, `Json`, `Finding`, `FindingsDoc`, `FlowStep`, `Severity`).

## Global Constraints

- Module path: `apps/taint`, Gradle path `:apps:taint`.
- Package root: `com.oversecured.sast.taint`.
- Analyzer-owned rule model package: `com.oversecured.sast.taint.model`.
- Rule model classes must have JavaBean getters compatible with the rules plan:
  - `RuleFile#getVersion()`, `RuleFile#getRules()`
  - `Rule#getId()`, `getVulnerabilityClass()`, `getSeverity()`, `getCwe()`, `getOwaspMobile()`, `getMessage()`, `getManifestConditions()`, `getSources()`, `getSinks()`, `getSanitizers()`, `getPropagators()`
  - `ManifestConditions#isReachableFromExported()`
  - `SourceSpec#getSignature()`
  - `SinkSpec#getSignature()`, `getTaintedArgs()`
  - `SanitizerSpec#getSignature()`
- YAML keys are snake_case (`vulnerability_class`, `owasp_mobile`, `manifest_conditions`, `reachable_from_exported`, `tainted_args`); model fields/getters are camelCase.
- Rule YAML may store bare FlowDroid-style signatures (`android.webkit.WebView: void loadUrl(java.lang.String)`) because the existing rules plan uses that shape. `apps/taint` must canonicalize both bare and bracketed signatures before using `SignatureParser.parse(...)`.
- CLI contract is exact: `taint --ast <ast-index> --facts <facts.json> --rule <rule.yaml> --out <findings.json>`.
- Library API writes deterministic output; stdout is only human progress text.
- Output must be `FindingsDoc` / `Finding` from `com.oversecured.sast.common`, with `analyzer = "taint-engine"`.
- `apps/taint` depends on `:common` and `:apps:parser`; it must not duplicate shared schemas.
- `apps/taint` may read rule files from `rules/` in tests, but rule files remain external assets and are not compiled into the module.
- Incomplete sanitizer handling is explicit: bypassable validation such as `host.endsWith("example.com")` must not kill taint; emitted findings include a note beginning `incomplete-sanitizer:`.
- Scope limits are contractual: do not add full IFDS/IDE, global alias analysis, reflection interpretation, native analysis, dynamic loading, or implicit flows while implementing this plan.
