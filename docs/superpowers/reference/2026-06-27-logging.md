# Logging Conventions

**Date:** 2026-06-27
**Status:** Required reference for implementation plans
**Audience:** agents and developers implementing individual Android Taint SAST pipeline modules
**Related:** [Shared Contracts and Naming Conventions](2026-06-27-shared-contracts-and-conventions.md) · [Error Handling](2026-06-27-error-handling.md)

This document is the source of truth for how every pipeline module emits logs. It exists so that independently implemented steps produce a consistent, greppable, low-noise log stream whether run standalone (CLI) or under the orchestrator.

---

## 1. Goals

- One logging facade and one log format across all modules.
- Logs emitted **only at boundaries** — deep internals stay silent.
- Each line is self-locating: you can tell **which function** and **what phase** from a glance or a `grep`.
- No secrets, no payload dumps, no double-logging.
- Stdout remains a non-contract (per shared-conventions §4.2); diagnostics go to the logger.

---

## 2. Backend and API

- **Facade:** SLF4J everywhere. Never call `System.out.println` / `System.err.println` for diagnostics.
- **Binding:** `org.slf4j:slf4j-simple` is the default for CLI/standalone runs (configured to write to `stderr`). The orchestrator/worker may put `logback-classic` on the runtime classpath instead — this is a runtime swap, no code change, because all code targets the SLF4J facade.
- **Logger declaration**, one per class:

  ```java
  private static final Logger log = LoggerFactory.getLogger(Decompiler.class);
  ```

- **Dependency:** `slf4j-api` is provided transitively; the binding is declared `runtimeOnly` in each runnable module's `build.gradle`. Versions come from the root `ext` block (`slf4jVersion`).

### 2.1 stdout vs logs

| Channel | Purpose | Contract? |
|---|---|---|
| **stdout** | human-facing success summary line from a CLI (`decompiled X -> Y`) | No — never parsed by another step |
| **stderr (SLF4J)** | all structured diagnostics (lifecycle, warnings, errors) | No — observability only |
| **artifacts** | the only data channel between steps | Yes |

A CLI may print exactly one human summary line to stdout on success; everything else is a log line.

## 3. Where we log — boundary logging

Logs are emitted **only at public boundary methods**. There are three altitudes:

| Layer | Logs? | Levels it may emit |
|---|---|---|
| **Outermost boundary** — CLI `call()`, Temporal activity impl | Yes | INFO start, INFO outcome, ERROR/WARN |
| **Module service boundary** — public entry method (e.g. `Decompiler.decompile`) | Yes | INFO start + INFO artifact-written; DEBUG sub-steps |
| **Deep helpers** — private methods, CFG nodes, taint worklist, parsers | **No** | — (return a value or throw) |

**The rule:** if you want to log inside a private helper, don't — return a value or throw, and let the boundary report it. This keeps the log volume proportional to *operations*, not *statements*, and guarantees one event is logged once.

**No log-and-rethrow.** A method that throws must not also log. Only the outermost boundary that *stops* the flow logs the failure (see [Error Handling](2026-06-27-error-handling.md) §boundary-translation).

---

## 4. What we log — levels

| Level | When | Who | Notes |
|---|---|---|---|
| **ERROR** | a failure that aborts the step | outermost boundary only | one line: typed exception message + cause; logged exactly once |
| **WARN** | recoverable / degraded outcome | any boundary | fail-soft skips (file jadx couldn't decompile), incomplete-sanitizer, empty result where output was expected; prefer one aggregated line with a count over N lines |
| **INFO** | lifecycle | boundaries | step start (key inputs), step success (result counts), artifact written |
| **DEBUG** | boundary sub-steps | boundaries | resolved paths, per-rule match counts; **off by default** |
| **TRACE** | reserved | — | unused for now (YAGNI) |

### 4.1 Never log

- Secrets, tokens, credentials.
- Full file contents or raw APK/dex bytes.
- Decompiled source bodies.

Log **metadata** instead: file paths, sizes, counts, durations, rule ids.

## 5. Emoji convention — function emoji + phase emoji

Every log line is formatted:

```
<function-emoji> <phase-emoji> <message>
```

### 5.1 Phase emoji (global, fixed across all modules)

| Emoji | Phase |
|---|---|
| ▶️ | start of a boundary operation |
| ✅ | success / completion |
| ❌ | error (aborting failure) |
| ⚠️ | warning / degraded |
| 📁 | artifact written |
| 🔍 | debug / inspection detail |

### 5.2 Function emoji (per boundary function, stable)

Each boundary function picks **one** emoji used on **all** of its log lines, for the life of the function. This lets you `grep` a single function's entire story. Every module's plan and README MUST carry an **emoji registry** table mapping `boundary function → emoji` to prevent collisions within the module. Reuse across modules is fine (modules are separate processes / logger namespaces).

### 5.3 Example (decompiler)

```
🚀 ▶️ decompile --apk Clone1.apk --out sources/
🔨 ▶️ decompiling Clone1.apk
🔨 📁 wrote 3 .java files + AndroidManifest.xml
🚀 ✅ decompiled Clone1.apk -> sources/
```

`grep 🔨` → everything `Decompiler.decompile` did. `grep ❌` → every aborting failure in the run.

### 5.4 Module emoji registry (seed)

Each module owns its table; this is the decompiler seed (extend per module):

| Module | Boundary function | Emoji |
|---|---|---|
| decompiler | `DecompilerCli.call` | 🚀 |
| decompiler | `Decompiler.decompile` | 🔨 |

## 6. Implementation checklist for step agents

Before implementing logging in a step:

- Declare one SLF4J `Logger` per class; no `System.out`/`System.err` for diagnostics.
- Log only at boundary methods (§3); keep deep helpers silent.
- Respect the level table (§4); never log secrets or payloads (§4.1).
- Prefix every line `<function-emoji> <phase-emoji>` (§5); register your function emojis in the module README/plan.
- One human summary line to stdout from a CLI on success is allowed; everything else is a log line.
- Failures are logged once, at the outermost boundary only (no log-and-rethrow).
